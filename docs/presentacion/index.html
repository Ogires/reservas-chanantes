RESERVAS CHANANTES · TFM02
03 · El problema

Las citas, todavía a mano.

  • Peluquerías, fisios, centros de estética… reservan por teléfono y WhatsApp.
  • Sin nada que evite solapamientos ni calcule la disponibilidad de forma fiable.
  • Sin cobro integrado: el pago se gestiona aparte, o no se gestiona.
  • Las alternativas comerciales son caras o demasiado genéricas para este perfil.
RESERVAS CHANANTES · TFM03
04 · La solución

Cada negocio, su página de reservas.

El negocio se registra, configura sus servicios y su horario, y publica una URL propia desde la que sus clientes reservan en tiempo real.

  • Panel /admin — servicios, horario, reservas y cobros
  • Página pública /[negocio] — disponibilidad y reserva
  • Portal del cliente /my — historial y autoservicio

RESERVAS CHANANTES · TFM07
08 · Arquitectura

Cuatro capas, una regla.

domainEntidades, objetos de valor y servicios de dominio puros.
applicationCasos de uso y puertos (interfaces de repositorio).
infrastructureAdaptadores Supabase, Stripe, Resend.
presentationNext.js: páginas, Server Actions, componentes.

La regla que lo sostiene todo: el dominio no importa ninguna dependencia de framework. No es declarativo — es verificable, y es lo que habilita una pirámide de pruebas de base ancha.

RESERVAS CHANANTES · TFM08
09 · Un problema difícil

Cero dobles reservas.
Lo garantiza la base de datos.

Dos clientes que piden el mismo hueco en el mismo instante son una condición de carrera clásica. En vez de resolverla con validaciones frágiles en la aplicación, la garantía vive en la capa autoritativa: una restricción de exclusión de PostgreSQL rechaza el solape a nivel físico.

EXCLUDE USING gist (tenant_id =, service_id =, time_range &&)
  WHERE status <> 'cancelled'
→ colisión = SlotTakenError (traducido en la frontera de infraestructura)

11 · La verdadera tesis
El producto es el vehículo.
La tesis es el método.

¿Se puede desarrollar software guiado por IA sin renunciar a la calidad de ingeniería? Para responderlo, el sistema se construye bajo disciplina explícita — Clean Architecture · DDD · TDD · SOLID — con trazabilidad en planes, revisiones y registro de desviaciones.

RESERVAS CHANANTES · TFM11
12 · Honestidad académica

Lo que aún falta, dicho de frente.

Limitaciones reconocidas

  • Monetización por plan diseñada, no activada.
  • Clean Architecture parcial en la rama de administración.
  • Defecto conocido de zona horaria.
  • Sin pruebas de integración contra una BD real (se usan dobles).

Ya subsanado + líneas futuras

  • RLS permisiva → acotada a propietario + titular (OWASP).
  • Añadidos CI/CD encadenado y E2E cross-browser.
  • Futuro: integración contra PostgreSQL efímero y modelado de amenazas.
